вҸә Root cause gefunden + Mechanismus code-verifiziert. Zusammenfassung:

  Warum die Embeddings fehlschlagen

  Befund: Zwei Probleme greifen ineinander вҖ” ein echter Batch-Overflow-Bug bei
  groГҹen Dokumenten, plus ein verschluckter Fehler, der genau diese Diagnose
  blockiert.

  1. Root Cause: embedMany batcht nach Anzahl, nicht nach Token-Budget

  Der Embedding-Aufruf lib/ingestion/embed.ts:23 nutzt AI-SDK embedMany.
  Verifiziert im SDK-Source:

  - embedMany splittet die Chunk-Liste per splitArray(values, 
  maxEmbeddingsPerCall) вҶ’ nur nach Item-Anzahl (OpenAI =В 2048),
  node_modules/ai/dist/index.js:11301.
  - OpenAI-Provider schickt dann bis zu 2048 Chunks in einem POST an
  /embeddings, node_modules/@ai-sdk/openai/dist/index.js:1922 вҖ” prГјft nur length
  > 2048, kein Token-Check.
  - OpenAI /embeddings lehnt jeden Request > 300.000 Token gesamt mit HTTP 400
  ab.

  Mechanismus (die Rechnung):
  - Chunker: 800 Token/Chunk (chunker.ts:42).
  - 300.000 / 800 вүҲ 375 Chunks pro Request = Obergrenze.
  - Dokument mit вүҘ ~375 Chunks (вүҲ langes PDF, Buch, ~150+ Seiten) вҶ’ ein einziger
  Request mit > 300k Token вҶ’ HTTP 400 вҶ’ embedMany wirft.
  - Kleine Quellen (Paste, Web, kurze PDFs) < 375 Chunks вҶ’ gehen durch.

  вҶ’ Exakt das Muster "mehrere Uploads fehlschlagen": die groГҹen kippen, die
  kleinen laufen. Size-abhГӨngig, deterministisch (400 вү  retrybar вҶ’ SDK-Retries
  helfen nicht).

  2. Warum du (und ich) die Ursache nicht siehst

  lib/ingestion/service.ts:381-385:
  try {
    embeddings = await this.deps.embedChunks(...)
  } catch {                                    // вҶҗ echter OpenAI-Fehler 
  weggeworfen
    throw new IngestionError(INGESTION_MESSAGES.embedFailed)
  }
  Leerer catch {}, kein console.error, echte OpenAI-Message verworfen. Jeder
  Embed-Fehler (400 Token-Limit, 429 Rate, 401 Key, Netz) kollabiert zur selben
  generischen "Embedding fehlgeschlagen вҖ” bitte erneut versuchen." Deshalb sieht
  error_message bei allen gleich aus und die eigentliche Ursache ist nirgends
  geloggt.

  Ausgeschlossen (verifiziert, nicht geraten)

  - Key fehlt/ungГјltig вҶ’ nein. Lokaler OPENAI_API_KEY getestet gegen
  /v1/embeddings: HTTP 200, 1536-dim. GГјltig + gedeckt.
  - Chunk zu groГҹ (>8191 Token) вҶ’ nein. Chunker kappt beiВ 800.
  - Quota lokal leer вҶ’ nein (200er ging durch).

  Ehrliche EinschrГӨnkung

  100%-BestГӨtigung, dass deine konkreten Fails der 300k-Fall sind (statt z.B.
  429 in Prod), geht nicht statisch вҖ” genau weil der echte Fehler verschluckt
  wird und lokales Supabase nicht lГӨuft (DB-Fehlerzeilen nicht abrufbar). Aber:
  Key funktioniert + "mehrere, nicht alle" + 300k-Overflow ist ein realer
  latenter Bug im Code вҶ’ das ist die mit Abstand belastbarste Ursache.

  Fix-Empfehlung (2 Schritte, nicht ungefragt angewandt)

  1. Sofort diagnosefГӨhig machen (3 Zeilen): catch (error) statt catch {},
  echten Fehler loggen + in error_message durchreichen. BestГӨtigt zugleich den
  Mechanismus.
  2. Eigentlicher Fix in embed.ts: embedMany-Batches nach Token-Budget bilden
  (z.B. вүӨ ~250k Token/Request via tiktoken-ZГӨhlung), statt SDK nach 2048 Items
  splitten zu lassen. Optional Retry/Backoff fГјr 429 beibehalten.

  Soll ich Schritt 1 (Un-Swallow, sofort) und/oder Schritt 2
  (Token-Budget-Batching) umsetzen? Schritt 1 ist minimal und macht die Ursache
  endgГјltig sichtbar.
