## Problem Statement

GoatbookLM hat den Core Loop (Notebooks, Ingestion, grounded Chat), aber keine Studio-Outputs. NotebookLM-Nutzer erwarten generierte Artefakte aus ihren Quellen. V2 Quick Wins in dieser Reihenfolge: **Reports** (dieses Doc, baubar), dann Flash Cards, Quiz; später ElevenLabs-Audio. Out of scope (voraussichtlich dauerhaft): Mindmap, Video, Slide Deck, Infographic.

Reports = generiertes Text-Dokument aus den Quellen des Notebooks in einem von **3 festen Formaten**: Briefing Doc, Study Guide, Blog Post. Keine KI-Suggestions, kein "Create Your Own" (bewusst gecuttet).

## What Makes This Cool

Das rechte Studio-Panel — bisher Platzhalter „Audio, Video & mehr — kommt bald" — wird real. Ein Klick, Report streamt live in den Viewer (NotebookLM-Feeling), bleibt als Artefakt in der Liste. Und: das Datenmodell trägt Flash Cards, Quiz und Audio ohne neue Migration — jedes Folge-Feature ist nur noch Prompt + Renderer.

## Constraints

- UI Deutsch, Design-System aus DESIGN.md (warm-neutral, cardless in Panels, Pastell nur für Kacheln, schwarze Primary-Pills).
- Parallel-Session baut core-loop-v2 (S0–S9) uncommitted im Haupt-Dir → dieses Feature maximal in **neuen Dateien**, Shell-Diff minimal (nur `StudioPanelBody`-Ersatz + Import).
- CLAUDE.md-Gates: RLS+revoke+grant in derselben Migration, `getUser()` server-seitig, Dialog statt Sheet, Delete mit Confirm-Dialog, `data-test` überall, tsc/lint/build grün.
- Kein Rate-Limiting in diesem Scope (existiert app-weit noch nicht, siehe TODOS.md — gilt für Chat genauso).

## Premises (bestätigt von Andi, 2026-07-20; P4 nach Review angepasst)

1. **Reports aktiviert das Studio-Panel.** Der Shell-Slot existiert bereits (`panel-placeholders.tsx` → `StudioPanelBody`), es wird nur der Body ersetzt — kein Layout-Umbau.
2. **EIN Datenmodell für alle Studio-Artefakte:** Tabelle `studio_artifacts` mit `type` (report | flashcards | quiz | audio), `status`, `title`, `content jsonb`. Folge-Features = neuer type + Prompt + Renderer.
3. **Generierung streamt** über einen Route Handler (Pattern von `app/api/chat/route.ts`, `streamText` + `claude-sonnet-5`); Persistenz server-seitig via `consumeStream()` + `after()` (läuft auch bei Client-Disconnect zu Ende). Kein synchroner Server-Action-Call. Audio später async (pgmq), gleiche Tabelle, gleicher Status-Lifecycle.
4. **Quellen-Scope v1 = alle `ready`-Quellen des Notebooks** mit nicht-leerem `content_text`. *Ursprünglich bestätigt war „ausgewählte Quellen (Checkboxen)" — der Selection-State existiert im committed Base aber nicht (Chat ragged übers ganze Notebook, keine Checkbox-Codepfade). Selection bauen hieße Bestandsdateien im Sources-Panel anfassen → maximale Merge-Konfliktfläche mit core-loop-v2. Deshalb: v1 ohne Auswahl; `source_ids` wird trotzdem als Snapshot persistiert, das API nimmt optional `sourceIds` entgegen (in v1 ignoriert) — Selection-Wiring kommt nach dem core-loop-v2-Merge ohne Schema- oder API-Bruch.*
5. **Merge-Risiko bewusst akzeptiert:** Worktree basiert auf committed Stand; core-loop-v2-Arbeit (Notes, S5-Panels) fehlt hier. Konfliktfläche klein halten = neue Dateien.

## Approaches Considered

### Approach A: Panel-Direkt (minimal)
3 Format-Kacheln direkt im Panel, kein Dialog, Streaming-Viewer, persist am Ende. Effort S. Schnellster Win, aber weicht vom NotebookLM-Flow ab, Aktionen (Rename/Delete/Copy) fehlen.

### Approach B: NotebookLM-Parität (GEWÄHLT)
„Create report"-Dialog mit 3 Format-Karten (wie Screenshot, minus Suggestions/Create-Your-Own), Studio-Panel mit Kachel + Artefakt-Liste, Viewer mit Breadcrumb (Studio → Report), Copy/Rename/Delete. Streaming live, persist am Ende. Effort M (~2-3h CC).

### Approach C: Reload-sicherer Hybrid
Wie B, plus server-seitige Chunk-Persistenz + Supabase Realtime → Reload-/Multi-Device-safe. Effort L. Für Text-Reports Overkill; Mechanik wird erst mit Audio nötig.

## Recommended Approach

**B.** Voller NotebookLM-Look bei kleinem Aufwand; Dialog + Kacheln sind exakt die Fläche, an die Flash Cards und Quiz andocken. C-Robustheit erst, wenn ElevenLabs-Audio kommt (dort Pflicht: Jobs > 60s, kein offenes HTTP-Fenster).

### Datenmodell (Migration `create_studio_artifacts`)

```sql
create table studio_artifacts (
  id uuid primary key default gen_random_uuid(),
  notebook_id uuid not null references notebooks(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  type text not null check (type in ('report','flashcards','quiz','audio')),
  format text check (format in ('briefing_doc','study_guide','blog_post')),
  title text not null,
  status text not null default 'generating'
    check (status in ('generating','ready','failed')),
  content jsonb,           -- report: {"markdown": "...", "truncated"?: true}; später: cards/questions/audio-ref
  source_ids uuid[] not null default '{}',  -- Snapshot der verwendeten Quellen
  error_message text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint report_requires_format check (type <> 'report' or format is not null)
);
```

Gleiche Migration: `enable row level security` + `revoke all` + `grant select,insert,update,delete to authenticated` + Policies (`auth.uid() = user_id` für alle 4 Operationen), Index auf `(notebook_id, created_at desc)`, `set_updated_at()`-Trigger (existiert bereits, siehe `20260719103134_create_core_schema.sql` — gleiche Anwendung wie bei notebooks/sources). Danach `supabase gen types typescript --local > lib/database.types.ts`.

`content`-Shape pro type (später): flashcards `{"cards":[{"front","back"}]}`, quiz `{"questions":[{"question","options":[4],"correct_index","explanation"}]}`, audio `{"storage_path","duration_s","voice"}`.

### Generierung (Route Handler `app/api/studio/generate/route.ts`)

- `export const runtime = "nodejs"`, `export const maxDuration = 120` (wie Chat-Route).
- POST, Zod-validiert als discriminated union: **Neu-Generierung** `{ notebookId, format, sourceIds? }` oder **Retry** `{ notebookId, retryArtifactId }` (bei Retry ist `format` im Body nicht erlaubt — kommt aus der Row). `sourceIds` optional und in v1 ignoriert (Forward-Compat für Selection). Auth: `supabase.auth.getUser()`, Notebook-Ownership prüfen; nicht gefunden/fremd → **einheitlich 404** (Chat-Konvention).
- Quellen laden: alle Quellen des Notebooks mit `status='ready'` und nicht-leerem `content_text`. 0 solcher Quellen → 422 mit Fehlermeldung (UI disabled das ohnehin).
- Row inserten (`status='generating'`, Titel provisorisch = Formatname, `source_ids` = geladene Quellen), **`artifactId` im Response-Header `X-Artifact-Id`**.
- Kontext-Budget: **~300k Zeichen gesamt** (≈ 140k Tokens bei ~2,1 Zeichen/Token für deutschen Text — Repo-Konvention aus `lib/chat/limits.ts`, dort empirisch verifiziert). Bei Überschreitung fair pro Quelle kürzen: gleiches Zeichen-Budget je Quelle, davon 70 % vom Anfang + 30 % vom Ende, Mitte schneiden (Marker `[… gekürzt …]`) — kein RAG-top-k, Reports brauchen Gesamtsicht.
- `streamText` mit `claude-sonnet-5` (wie Chat), `maxOutputTokens 8192`. Response via `toTextStreamResponse()` — reiner Text-Stream, kein `useChat`-Protokoll (Viewer ist kein Chat). Prompt pro Format (fest Deutsch, Markdown mit H2-Abschnitten; Briefing: Key Insights + Zitate; Study Guide: Kurzfragen + Essay-Fragen + Glossar; Blog Post: lesbarer Artikel mit These). Erste Zeile der Antwort = Titel (Konvention `# <Titel>`), wird beim Persistieren extrahiert.
- Persistenz: `result.consumeStream()` + **Persist in `after()`** (Pattern der Chat-Route; auf Vercel garantiert nur `after()` Post-Response-Ausführung — kein `onFinish`-Callback): in `after()` `await result.text` + `finishReason`, dann Row-Update `status='ready'`, `content={"markdown":...}`, `title` aus H1. Läuft server-seitig zu Ende, auch wenn der Client den Tab schließt. `finishReason !== 'stop'` → trotzdem `ready`, aber Hinweiszeile ans Markdown angehängt (Chat-Pattern) + `content.truncated=true`. Fehler → `status='failed'` + `error_message`.
- Titel-Konvention: beginnt die Antwort mit `# <Titel>`, wird die H1 als `title` extrahiert und **aus dem persistierten Markdown gestrippt** (Viewer zeigt Titel separat — sonst doppelt). Der Viewer strippt die erste `# `-Zeile auch im Live-Stream (zeigt sie als Titel-Header). Keine H1 in der Antwort → provisorischer Formatname bleibt `title`.
- Backstop (nur Server-Crash): Rows mit `status='generating'` und **`updated_at`** > 5 min alt (nicht `created_at` — Retry setzt `updated_at` via Trigger neu) werden in der UI als fehlgeschlagen angezeigt; Retry-fähig (s.u.). Kein Cron.

### Leerer Zustand / Status / Retry

- Keine Artefakte: Kachel + kurzer Hinweistext statt bisherigem Platzhalter.
- `generating`: Skeleton-Row in Liste (wie NotebookLM „Loading…"-Karte) bzw. Live-Stream im Viewer.
- `failed` **oder** `generating` älter als 5 min (Backstop): Row mit Fehler-Badge + „Erneut versuchen".
- **Retry = dieselbe Row wird wiederverwendet:** Route-Variante `{ notebookId, retryArtifactId }` setzt die Row zurück auf `generating` (Content/Fehler geleert) und generiert mit dem persistierten `format` neu. Quellen werden dabei **frisch geladen** (aktuelle ready-Quellen, nicht der alte Snapshot — inzwischen ingestete Quellen sollen rein), `source_ids` wird mit dem neuen Stand überschrieben. Kein Delete+Insert, keine Duplikate.
- **Retry-Guard (Race-Schutz):** Reset nur als konditionales Update — `where status='failed' or (status='generating' and updated_at < now() - interval '5 minutes')`. 0 Rows getroffen → 409 (zweiter Tab/Doppelklick clobbert keine laufende Generierung).

## Merge- & Parallelisierungs-Plan

Ziel: Sobald die Studio-Basis steht, können Flash Cards, Quiz und Audio in parallelen Sessions gebaut werden.

**Studio-Basis =** Migration `studio_artifacts` + Studio-Panel (Kachel/Liste/Viewer-Gerüst) + Generate-Route (generisch gedacht: type-Parameter, Prompt-Registry) + Reports als erste Ausprägung. Genau der Scope dieses Docs.

**Ablauf:**
1. Basis + Reports fertig auf `feat/studio-quick-wins` (Gates grün, committed).
2. Merge zurück nach `feat/core-loop-prep` — Voraussetzung: die core-loop-v2-Session hat ihren Stand committed (Konfliktfläche: `notebook-detail-shell.tsx` / `panel-placeholders.tsx`, erwartbar klein, da hier nur Panel-Body-Ersatz). Nach dem Merge: Selection-Wiring (P4) wird möglich.
3. Folge-Features je in eigenem Worktree/Branch ab dem gemergten Stand:
   - `feat/studio-flashcards` — type `flashcards`, structured output, Karten-Renderer (Flip-UI)
   - `feat/studio-quiz` — type `quiz`, structured output, Quiz-Renderer (Antwort-Check + Erklärung)
   - `feat/studio-audio` — type `audio`, ElevenLabs, pgmq-Job + Storage + Player (braucht C-Mechanik: Status-Polling/Realtime)
4. Falls Merge nach `feat/core-loop-prep` blockiert (Session mitten in Arbeit): Parallel-Branches ersatzweise direkt ab `feat/studio-quick-wins` ziehen — die Studio-Basis ist dort vollständig.

**Damit Parallel-Sessions konfliktfrei bleiben:** jedes Folge-Feature fasst nur eigene neue Dateien an (Renderer-Komponente, Prompt-Eintrag) + je einen Registry-Eintrag (Prompt-Registry, Renderer-Switch, Kachel-Liste). Registry-Dateien klein und append-only halten.
