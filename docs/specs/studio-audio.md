## Problem Statement

Letztes Studio-Quick-Win und das einzige mit Tiefgang (Andis Vorgabe): **Audio Overview** wie NotebookLM — ein generierter Podcast/Audio-Beitrag aus den Quellen, mit vollem Customize-Dialog: **Format** (Deep Dive / Brief / Critique / Debate), **Sprache**, **Länge** (Kurz/Standard/Lang), **Quellen-Auswahl**, **Fokus-Freitext** („Worauf sollen die Hosts eingehen?"). TTS über **ElevenLabs** (strategischer Fit zu Everlast).

## What Makes This Cool

Klick auf „Audio" → Customize-Dialog → Minuten später liegt ein gesprochener Zwei-Personen-Deep-Dive über die eigenen PDFs im Studio-Panel, mit Player und Transkript. Die Phasen-Pipeline zeigt dabei echten Fortschritt („Skript wird geschrieben…" → „Audio wird erzeugt… 12/28") statt eines blinden Spinners.

## Constraints

- UI Deutsch, DESIGN.md-System; Kachel-Grid im Studio-Panel wird 2×2 (4. Kachel `--card-4`).
- Parallel-Session-Regel bleibt: neue Dateien bevorzugt; angefasste Bestandsdateien nur eigene Studio-Dateien + minimal Route/Schema.
- CLAUDE.md-Gates (RLS+grants in einer Migration, fail-closed Auth, `data-test`, Dialog-Konvention, tsc/lint/build).
- `ELEVENLABS_API_KEY` server-only; fehlt der Key → Create-Route antwortet 503 mit klarer Meldung (fail-closed, kein stiller Mock). Andi trägt den Key in `.env.local` ein (D1-Antwort).
- Kosten bewusst: jede Generierung kostet ElevenLabs-Credits + Claude-Tokens; kein Rate-Limit in diesem Scope (TODOS.md, vor Public Launch).

## Premises (bestätigt von Andi, 2026-07-20, D1)

1. **Async via pgmq**: eigene Queue `studio_audio` + eigene Worker-Route `app/api/studio-worker/route.ts` (`maxDuration 300`, `runtime nodejs`), getriggert via pg_cron + pg_net alle 15s, Shared-Secret aus eigener Config-Tabelle — 1:1 das Muster von `app/api/ingestion-worker/route.ts` / `20260719144042_create_ingestion_queue.sql`. Kein 120s-HTTP-Pfad.
2. **Zweistufig**: Claude (`claude-sonnet-5`) generiert ein Format-abhängiges Skript (structured output), ElevenLabs spricht es. Volle Kontrolle über Format/Fokus/Narrative liegt bei unserem Prompt.
3. **Neuer privater Storage-Bucket `studio-audio`**, Pfad `{user_id}/{artifact_id}.mp3` (+ `{user_id}/{artifact_id}/segments/{i}.mp3` als Zwischenstand), Owner-Policies auf `storage.objects`, Playback via Signed URL.
4. **Customization exakt wie der NotebookLM-Screenshot**: Format, Sprache (Default Deutsch), Länge, Quellen (bestehender `SourcePicker`), Fokus-Textarea. Keine weiteren Optionen (keine Voice-Auswahl in v1).
5. **Kosten bewusst akzeptiert** (siehe Constraints).

## Approaches Considered

### Approach A: Ein-Schuss-Worker (minimal)
Ein Tick macht Skript + TTS + Concat + Upload; Fehler = ganzer Job failed, Retry zahlt alles doppelt. Effort M. Verworfen: blinder Spinner über Minuten, doppelte Skript-Kosten bei jedem Retry.

### Approach B: Phasen-Job mit Zwischenstand (GEWÄHLT, D2)
Phasen (`script` → `tts` → fertig) in der Artefakt-Row persistiert; Retry und Folge-Ticks setzen an der letzten Phase an; Panel zeigt Phase + TTS-Fortschritt. Effort L (~4-5h CC).

### Approach C: Ein Text-to-Dialogue-Call
Ganzes Skript in einem ElevenLabs-Call. Beste Übergänge, aber Längen-Limits unverifiziert — als **TTS-Strategie innerhalb von B** beim Bau geprüft (kurze Formate ein Call, lange per-Turn), nicht als eigener Ansatz.

## Recommended Approach — Detail

### Migration 2: `create_studio_audio_infra`

Analog `20260719144042_create_ingestion_queue.sql`:
- `select pgmq.create('studio_audio');`
- Enqueue-RPC `public.enqueue_studio_audio_job(p_artifact_id uuid)` — `security definer`, **grant execute nur an `service_role`** (revoke von authenticated/anon), Payload `{"artifact_id": ...}`.
- Read/Delete-RPCs analog den bestehenden pgmq-Wrappern (`readIngestionJobs`-Muster in `lib/ingestion/queue.ts`), `vt = 600`.
- Config-Tabelle `public.studio_worker_config` (eine Row: `worker_url`, `worker_secret default gen_random_uuid()`): RLS enabled, revoke all, **null Policies** — gelesen ausschließlich vom Worker über den Admin-Client (BYPASSRLS), exakt wie `ingestion_worker_config`.
- `cron.schedule('studio-audio-worker', '15 seconds', ...)` mit pg_net-POST auf `worker_url` + `x-worker-secret`-Header — exakt das Ingestion-Muster (lokale URL via seed/config).
- Storage-Bucket `studio-audio`: `insert into storage.buckets (id, name, public, file_size_limit, allowed_mime_types) values ('studio-audio','studio-audio', false, 52428800, array['audio/mpeg']);` + Owner-Policies auf `storage.objects` — **nur `select` und `delete`** für authenticated (Pfad-Präfix `auth.uid()`). KEINE insert/update-Policies: Uploads macht ausschließlich der Worker mit Admin-Client; User-Uploads in den Bucket wären reine Missbrauchsfläche (Review-Fix R1-6, konsistent mit dem sources-Bucket, der update ebenfalls bewusst weglässt).

Danach Types regenerieren (Worktree-Konvention: `docker exec supabase_db_goatbooklm psql` + History-Insert, dann `supabase gen types` und fremde Tabellen rausschneiden — siehe Memory).

### content-jsonb-Lifecycle (`type='audio'`)

```
Create:   { params: {language, length, focus}, phase: "script" }
Skript:   + script: {title, turns: [{speaker: 1|2, text}]}, phase: "tts",
          tts: {done: 0, total: N}
TTS:      tts.done zählt pro fertigem Segment hoch (Row-Update je Turn)
Fertig:   { params, script, storage_path, phase: "done" } · status='ready',
          title = script.title, Segmente im Storage gelöscht
Fehler:   status='failed' + error_message; phase/script BLEIBEN erhalten
```

`storage_path` ist KEINE Spalte — es lebt ausschließlich im `content`-jsonb (Review-Fix R1-9). Der veraltete content-Shape-Kommentar in `20260720210000_create_studio_artifacts.sql` ist ein `--`-Datei-Kommentar (nachträglich nicht editierbar); Migration 1 setzt stattdessen ein `comment on column public.studio_artifacts.content` mit dem tatsächlichen Audio-Lifecycle als kanonischer Doku.

**Retry-Anpassung (Service):** `claimRetry` leert heute `content` — für `type='audio'` darf es das NICHT (Phasen-Zwischenstand + bezahltes Skript). Anpassung:
- `claimRetry` bekommt `preserveContent: boolean` (Audio: true) — dann bleibt `content` unangetastet; gesetzt werden `status='generating'`, `error_message=null` und (phasenabhängig, siehe unten) `source_ids`. Danach re-enqueue.
- **Stale-Fenster per Typ** (Review-Fix R1-2): `claimRetry` nutzt für Audio dasselbe 15-min-Fenster wie die UI (geteilte Konstante `STALE_GENERATING_MINUTES_AUDIO = 15`), nicht die 5 min der Inline-Typen — sonst könnte ein Retry in der legalen Tick-Lücke (Budget-Exit wartet bis ~10 min auf vt-Ablauf) einen laufenden Job claimen → doppelte pgmq-Message, doppelte TTS-Kosten.
- **Quellen bei Retry** (Review-Fix R1-10): `phase === 'script'` (Skript war noch nicht fertig) → `source_ids` frisch laden wie beim Nicht-Audio-Retry (Schnittmenge, sonst alle). `phase === 'tts'` → Snapshot bewusst behalten, das bezahlte Skript basiert darauf.

### Skript-Generierung (Phase `script`, im Worker)

- `generateObject` mit Schema `{ title, turns: [{speaker: z.union([z.literal(1), z.literal(2)]), text }] .min(3).max(60) }`; Critique erzwingt per Prompt nur Sprecher 1.
- **Kosten-Cap** (Review-Fix R1-4): vor Eintritt in die TTS-Phase Gesamt-Zeichen des Skripts prüfen — Limit **30.000 Zeichen** (≈ 20 min Audio, deckt „Lang" mit Puffer). Drüber → `failed` mit Meldung „Skript zu lang geraten — bitte erneut versuchen" (kein unbegrenzter ElevenLabs-Spend durch Runaway-Generierung).
- System-Prompt pro Format (Registry `lib/studio/audio-prompts.ts`, append-only):
  - **deep_dive**: 2 Hosts, lebhaftes Gespräch, Themen der Quellen verbinden und vertiefen; Host 1 führt, Host 2 fragt nach/ergänzt.
  - **brief**: 2 Hosts, kompakter Überblick über die Kernideen, kein Abschweifen.
  - **critique**: 1 Experten-Stimme, konstruktives Review des Materials (Stärken, Schwächen, konkrete Verbesserungen).
  - **debate**: 2 Hosts mit bewusst gegensätzlichen Positionen aus den Quellen, fair und beleuchtend.
- Sprache: `params.language` (Dropdown-Werte: de, en, es, fr, it, pt, hi, ja — von `eleven_multilingual_v2` abgedeckt); Skript wird komplett in dieser Sprache geschrieben.
- Länge (gesprochen ~150 Wörter/min): kurz ≈ 500-700 Wörter (~4 min), standard ≈ 1.300-1.600 (~10 min), lang ≈ 2.200-2.800 (~15-18 min). `maxOutputTokens 16000` (deutsch ~2,1 Zeichen/Token — lang braucht Luft; Wert beim Bau gegen Modell-Limit verifiziert).
- Kontext: `buildSourcesBlock` (300k-Budget) über die `source_ids` der Row; `params.focus` als zusätzliche Weisung im User-Turn.
- Persist: `script` + `phase: "tts"` + `tts.total = turns.length` in einem Update. Tick endet danach (Skript-Phase und TTS-Phase nie im selben Tick erzwungen — Zeitbudget).

### TTS (Phase `tts`, im Worker)

- **Primär beim Bau prüfen:** ElevenLabs **Text-to-Dialogue-API** (v3) — ein Call, ein File. Wenn verfügbar und das Skript unter dem Request-Limit liegt → nutzen, `storage_path` direkt schreiben. Verifikation gegen die aktuelle API-Doku ist Teil des Baus, nicht dieser Spec.
- **Fallback (sicherer Pfad, wird auf jeden Fall gebaut):** pro Turn `POST /v1/text-to-speech/{voice_id}` (REST via fetch, kein SDK-Zwang), `model_id: eleven_multilingual_v2`, `output_format: mp3_44100_128`, mit `previous_text`/`next_text` des Nachbar-Turns für Prosodie-Kontinuität. Voices: `ELEVENLABS_VOICE_HOST1`/`_HOST2` (env, Defaults = zwei multilingual Premade-Voices, beim Bau festgelegt).
- Jedes Segment nach `{user_id}/{artifact_id}/segments/{index}.mp3` (Admin-Client-Upload, upsert — idempotent bei Tick-Wiederholung), `tts.done` hochzählen.
- **Zeitbudget-Guard:** nach jedem Segment prüfen, ob > 240s Tick-Zeit verbraucht → Tick beendet OHNE den pgmq-Job zu löschen; `vt=600` läuft ab, der nächste Tick liest denselben Job und setzt bei `tts.done` fort (Segmente sind persistiert). Idempotenz über upsert + done-Zähler.
- Abschluss: Segmente in Reihenfolge herunterladen, MP3-Buffer konkatenieren — **von Segment 2..n dabei ID3v2-Header + Xing/Info-Frame strippen** (Review-Fix R1-7: mittendrin liegende Header sind decoder-abhängig, und der Info-Frame des ersten Segments würde sonst die Gesamtdauer im `<audio>`-Element falsch reporten / Seeking brechen; Segment 1 behält seinen Header). Gleicher Encoder/Bitrate (mp3_44100_128) → Frame-Concat Player-kompatibel, kein ffmpeg auf Vercel. Upload `{user_id}/{artifact_id}.mp3`, Segmente löschen, Row → `ready` (`content.storage_path`, `title`), pgmq-Job löschen.
- Jeder Fehler: Row → `failed` + verständliche deutsche `error_message` (ElevenLabs-Fehler gemappt: 401 → „ElevenLabs-Key ungültig", 429 → „ElevenLabs-Kontingent erschöpft", sonst generisch), Job löschen (Retry läuft über die UI, nicht über pgmq-Redelivery).

### Worker-Route `app/api/studio-worker/route.ts`

Kopie des Ingestion-Worker-Gerüsts: `maxDuration 300`, Secret aus `studio_worker_config` (TTL-Cache, `timingSafeEqual`, fail-closed), `READ_BATCH_SIZE = 1` (ein Audio-Job pro Tick — Audio-Jobs sind schwer; Ingestion-Queue bleibt unberührt getrennt). Dispatch nach `content.phase`. Reine Orchestrierung in `lib/studio/audio-worker.ts` (pure, Deps injiziert: db-Admin, `generateScript`, `synthesizeTurn`, storage — testbar mit Mocks).

**Message-Guard** (Review-Fix R1-3, Analog zum `SourceNotFoundError`-Pfad in `lib/ingestion/worker.ts`): jeder Tick lädt zuerst die Artefakt-Row. Row nicht gefunden (Artefakt gelöscht während Job queued/in-flight) ODER `status !== 'generating'` (bereits ready/failed — z. B. Retry-Re-Enqueue trifft auf Redelivery eines abgestürzten Ticks) → **Message löschen, keine Arbeit, kein Fehler.**

### Create/Retry (bestehende Route `app/api/studio/generate`)

- Schema-Union um Audio-Variante erweitert: `{ notebookId, type: 'audio', format: deep_dive|brief|critique|debate, language, length: kurz|standard|lang, focus?: string(max 500), sourceIds? }`.
- Route: Key-Gate (`ELEVENLABS_API_KEY` fehlt → 503), Quellen laden wie gehabt, Row insert (`phase: "script"` in content, `format` in Spalte), **enqueue via Admin-Client-RPC** (Muster `processSourceAction`), Response 202 + `artifactId`.
- Retry-Zweig: `type==='audio'` → `claimRetry(preserveContent: true)` + re-enqueue statt Inline-Generierung. Audio-422-Sonderfall („kann hier nicht neu erstellt werden") entfällt damit.

### UI

- **4. Kachel „Audio"** (`--card-4`), Kachel-Grid `grid-cols-2` (2×2) statt `grid-cols-3`.
- **CreateArtifactDialog, Audio-Zweig** (Screenshot-Parität): 4 Format-Karten (Deep Dive „Ein lebhaftes Gespräch zweier Hosts…", Brief, Critique, Debate — deutsche Beschreibungen), Sprache-Select (nativer `<select>` styled oder shadcn, Default Deutsch), Länge als 3er-Segment (Kurz/Standard/Lang, Default Standard), Fokus-`Textarea` mit „Dinge zum Ausprobieren"-Platzhalter (Quelle fokussieren / Thema fokussieren / Zielgruppe), `SourcePicker`. `data-test` überall.
- **Listen-Row generating**: Phasen-Text aus `content`: „Skript wird geschrieben…" bzw. „Audio wird erzeugt… {done}/{total}" (Panel-Poll liefert die Row alle 2s). Stale-Backstop: Audio-Jobs dürfen länger laufen → Stale-Fenster für `type='audio'` = **15 min** (nicht 5) — `updated_at` wird ja pro Segment-Update frisch.
- **AudioViewer** (`audio-viewer.tsx`): Breadcrumb „Studio › Audio", Titel, `<audio controls>` mit Signed URL (Browser-Client `storage.from('studio-audio').createSignedUrl(...)` — RLS-Policies scopen auf Owner), Download-Link (gleiche URL), darunter **Transkript** aus `content.script.turns` (Sprecher-Label „Host 1/Host 2" bzw. „Expertin"). Kebab Umbenennen/Löschen wie gehabt.
- **Delete räumt auch Storage — in der L1-Reihenfolge** (Review-Fix R1-1): Pfade aus der Row lesen → **Row löschen → dann Storage-Objekte entfernen** (final + evtl. Segmente). Genau das Muster aus `app/(app)/notebooks/actions.ts` (`getNotebookPdfStoragePaths` → DB-Delete → `deleteStorageObjects`): ein fehlgeschlagener DB-Delete darf nie eine Row hinterlassen, die auf eine schon gelöschte Datei zeigt. Storage-Fehler nach erfolgreichem Row-Delete = geloggt, nicht fatal (Orphan-Objekt statt kaputter Row).
- **Notebook-Delete** (Review-Fix R1-5): `deleteNotebookAction` räumt heute nur den `sources`-Bucket. Erweiterung: zusätzlich alle `studio-audio`-Pfade des Notebooks einsammeln (aus den `studio_artifacts`-Rows vor dem Cascade-Delete) und nach dem Row-Delete entfernen — gleiche L1-Reihenfolge.

- [ ] Migrationen: Format-Constraints erweitert; Queue + RPCs (service_role-only) + Config-Tabelle + cron.schedule + Bucket mit Owner-Policies — RLS/revoke/grant jeweils in derselben Migration; Types regeneriert.

- [ ] `ELEVENLABS_API_KEY` fehlt → Create antwortet 503 mit deutscher Meldung; nichts wird enqueued.
- [ ] Worker fail-closed (Secret-Header, constant-time), verarbeitet Phasen; Skript-Phase persistiert `script` + `tts.total`; TTS-Phase zählt `tts.done` sichtbar hoch.
- [ ] Tick-Budget-Guard: Job überlebt Tick-Grenze (Segmente persistiert, Fortsetzung im Folge-Tick, keine doppelten Segmente durch upsert).
- [ ] Fertiges Artefakt: eine MP3 im Bucket, Segmente gelöscht, `ready`, Titel aus Skript; Player spielt via Signed URL; Transkript sichtbar.
- [ ] Fremdes Notebook → 404; Retry auf laufenden Job → 409 (Audio-Stale-Fenster 15 min, geteilte Konstante UI+Service); Audio-Retry behält `script` (kein doppelter Claude-Call, wenn Skript-Phase fertig war); Retry in Skript-Phase lädt Quellen frisch.
- [ ] Worker-Message-Guard: gelöschtes/bereits-finalisiertes Artefakt → Message weg, keine Arbeit.
- [ ] Skript-Kosten-Cap: > 60 Turns oder > 30k Zeichen → `failed`, kein TTS-Call.
- [ ] Delete entfernt Row UND Storage-Objekte in L1-Reihenfolge (Row zuerst); Notebook-Delete räumt auch `studio-audio`-Pfade.
- [ ] Bucket-Policies: authenticated nur select+delete, kein insert/update.
- [ ] ElevenLabs-Fehler (401/429/5xx) → `failed` mit gemappter deutscher Meldung + Retry-Button.
- [ ] Unit-Tests: audio-worker-Orchestrierung mit gemockten Deps (Phasen-Übergänge, Budget-Guard, Idempotenz); Prompt-/Schema-Tests.
- [ ] E2E (mit echtem Key, sonst `test.skip`): Audio „Brief/Kurz/Deutsch" erstellen → Phasen-Badge sichtbar → `ready` ≤ 300s → Player mit `src` + Transkript → löschen (Row + Storage weg).
- [ ] `pnpm tsc --noEmit`, `pnpm next lint`, `pnpm next build` grün; volle E2E-Suite grün.
